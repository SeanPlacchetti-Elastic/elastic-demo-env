Guided Investigations — Pivot Through Your Data

Every 10 minutes, a cascading degradation ripples through the development microservices. Use these narratives to practice the core Elastic Observability workflow: detect → triage → correlate → root-cause.

Investigation 1: The Cascading Slowdown
Symptom: APM Service Map shows red/yellow latency indicators across multiple services.

Steps:
1. Open Observability → APM → Service Map. Watch for services turning yellow/red.
2. Click the first degraded service (usually east-catalog). Open Transactions.
3. Click a slow transaction → Trace waterfall. See which downstream call took longest.
4. Actions → View in Discover. Pivot to logs filtered by trace.id.
5. Look for scenario=degraded and "database connection pool exhausted".
6. Open ML → Anomaly Explorer. The latency job flags the same window.

Root cause: Catalog degrades at second 420 of each 10-minute cycle. Other services cascade: inventory (440s), pricing (450s), reviews (460s), orders (470s), recommendations (480s).
Investigation 2: Error Spike Correlation
Symptom: ML error rate job flags an anomaly. Multiple services show 503 errors simultaneously.

Steps:
1. Open ML → Anomaly Explorer. Select demo-apm-error-rate. Note flagged services.
2. Click anomaly swimlane → View in APM. The Errors tab shows specific exceptions.
3. Compare error messages: catalog says "db pool exhaustion", orders says "deadlock" — different symptoms, same window.
4. Logs: filter log.level: error AND scenario: degraded. All errors cluster in 90 seconds.

Root cause: Each service returns 503s at different rates (10-15%). Errors are correlated in time but differ per language — the pattern of a real cascading failure.
Investigation 3: The Latency Outlier
Symptom: demo-latency-outliers ML job detects one service with mean latency 10x above baseline.

Steps:
1. ML → Anomaly Explorer → demo-latency-outliers. Partition view shows the outlier.
2. east-recommendations (Rust) has the highest score — 600-2000ms in degraded mode.
3. APM → Transactions. Latency distribution shows bimodal: 10-80ms (normal) vs 600-2000ms (degraded).
4. Open a slow trace. Waterfall shows recommendation computation dominating duration.
5. Pivot to Logs. See "Recommendation model timeout".

Root cause: Recommendations degrades last (second 480) but has the most extreme latency, making it the biggest outlier.

What this demo covers

  • Two-cluster architecture — East is the production cluster where all observability data lives. West is the operations/NOC cluster that reads east data via CCS and hosts alerting rules.
  • Cross-Cluster Search (CCS) — West Kibana queries cluster-east:* indices. Three cross-cluster data views: APM traces, Filebeat logs, Heartbeat uptime.
  • Cross-Cluster Alerting — Three rules in west Kibana watch east production data: high error rate, degraded mode detection, health check failures.
  • APM & Distributed Tracing — All services ship traces to east. Distributed traces span 7 languages with up to 4-hop depth. ML anomaly detection runs on APM data.
  • Dual Environments — Development (port 8001, anomalies enabled) and Production (port 8002, stable). Same service names, different ELASTIC_APM_ENVIRONMENT. APM data streams split by namespace.
  • Application Search — Dedicated search-es01 cluster powers lexical search, search-as-you-type, and completion suggester for the product catalog.
  • Synthetic Monitoring — Fleet Server + elastic-agent-complete on east. Three Playwright browser journeys test real user flows.
  • Heartbeat — Monitors both clusters and all microservice health endpoints. Data ships to east.
  • Filebeat + Metricbeat — East filebeat autodiscovers container logs. East metricbeat collects ES, Kibana, and Docker metrics. West metricbeat: stack monitoring only.
