Guided Investigations — Pivot Through Your Data

Every 4 minutes, a cascading degradation ripples through the development refueling ops services. Use these playbooks to practice the core Elastic Observability workflow: detect → triage → correlate → root-cause. All investigations run against East Kibana (port 5602) unless otherwise noted.

Investigation 1 — The Cascading Slowdown Platinum
Scenario: Mission coordinator notices the ops portal is slow. Multiple services are degrading but no single alert has fired yet.
  1. Open Observability → APM → Service Map. Wait for the degradation window — services turn yellow then red in cascade order starting around second 168 of the 4-minute cycle.
  2. Click east-catalog (Aircraft Registry) — it degrades first. Select Transactions, filter to GET /aircraft. Watch P95 latency spike from ~50ms to 800–1500ms.
  3. Click a slow transaction (sort by duration descending). Open the Trace waterfall. Confirm that 60–80% of total duration is inside the east-catalog span, not downstream calls.
  4. In the waterfall header click View in Discover. Discover opens pre-filtered to trace.id: <id>. You'll see log entries with scenario: degraded and the message "aircraft registry database connection pool exhausted".
  5. Back in APM, click through east-inventory, east-pricing, and east-orders. Each shows a degradation window starting progressively later (176s, 180s, 188s). This is the cascade.
  6. Open Machine Learning → Anomaly Explorer → select demo-latency-outliers. All six services appear in the swimlane. The anomaly scores align precisely with the windows in the reference table below.
Key finding: The Service Map shows the blast radius. The trace waterfall localises the root service. Logs confirm the failure mode. ML confirms it's statistically anomalous — not noise.
Root cause: east-catalog exhausts its DB connection pool at second 168. Every dependent service (all six) begins returning 503s as its upstream catalog calls time out. The stagger is architectural — each service has a different retry/timeout budget.
Investigation 2 — Error Spike Correlation Platinum Enterprise
Scenario: An ML anomaly fires for the error rate job. Multiple services show 503s simultaneously — is this one failure or six independent incidents?
  1. Open Machine Learning → Anomaly Explorer → select demo-apm-error-rate. The swimlane shows multiple services flagged in the same 5-minute bucket. Anomaly scores will be 50–95 depending on cycle position.
  2. Click an anomaly cell and choose View in APM. The Errors tab opens for that service. Note the exception message — each language reports the failure differently: Java says "db pool exhaustion", Ruby says "database deadlock", Go says "depot sync timeout".
  3. Open Observability → Logs → Explorer (or Discover against filebeat-*). Run the KQL: log.level: (WARN OR ERROR) AND scenario: degraded. All matching log lines cluster within the same 36-second degradation window.
  4. Add service.name as a breakdown field. You'll see that all six services emit degraded errors in the same time bucket — different messages, same timestamp cluster. That pattern rules out independent failures.
  5. Now open Observability → Logs → Log Anomalies. The demo-log-rate ML job surfaces the same window as an unusual log volume spike, corroborating the APM error rate anomaly from a different signal.
  6. Check Observability → SLOs. Filter to the affected services — their availability SLOs (99.5% target) will be burning down during the degradation window. The error budget remaining chart shows the impact rate.
Key finding: Different error messages across services are a signature of cascading failure, not independent bugs. If it were six separate bugs, the timestamps would not align this precisely.
Root cause: All 503 errors are downstream consequences of the catalog connection pool exhaustion. The polyglot error messages describe each service's local failure response, not the underlying cause — a common trap in microservice debugging.
Investigation 3 — The Latency Outlier Platinum Enterprise
Scenario: SLO burn rate alert fires for Tanker Pairings. Latency is far outside normal bounds, but error rate looks almost normal. What's happening?
  1. Open Machine Learning → Anomaly Explorer → demo-latency-outliers. Switch to Single Metric Viewer and select partition east-recommendations. The model bound shows a normal baseline of 10–80ms. Actual values during degradation are 600–2000ms.
  2. Open Observability → APM → Services → east-recommendations → Transactions. The Latency distribution histogram shows a bimodal shape — two distinct peaks, not a long tail. This is a classic on/off degradation pattern, not gradual drift.
  3. Filter transactions by duration > 500ms. Open one. The trace waterfall shows the entire request time is consumed inside a single recommendations span — no downstream calls. The slowness is internal to the service.
  4. In the waterfall click View in Discover. Filter to that trace.id and look for the log message containing "Recommendation model timeout". Note the transaction.id in the log matches the trace.
  5. Now compare with a normal trace (duration < 100ms) from the same service. The same endpoint, same code path — but without the timeout. The bimodal distribution is the on/off degradation toggle, not a load-dependent curve.
  6. Open Observability → SLOs → east-recommendations latency SLO (target: P95 < 500ms). During the degradation window, P95 blows past 500ms — the SLO is actively breaching and burning error budget.
Key finding: The bimodal latency distribution is a critical diagnostic signal. It means the problem is a hard mode-switch (degraded vs. normal), not load-induced — narrowing root cause significantly.
Root cause: The Rust recommendation engine simulates a model inference timeout during its degradation window (second 192–204). It's the last service to degrade and has the most extreme latency impact — but the lowest error rate, making it the hardest to catch without ML.
Investigation 4 — Cross-Cluster NOC View Basic Platinum
Scenario: You're an ops engineer on the NOC team. You have read-only visibility into the production east cluster via CCS from your west cluster — no direct production access required.
  1. Open West Kibana (port 5601) and log in as ccs-analyst / CcsAnalyst1!. This account has cross-cluster read access but cannot modify east cluster data.
  2. Open Observability → APM. The data view is cluster-east:traces-apm*. You're seeing east production APM data through CCS — rendered in your local Kibana without any data replication.
  3. Open Alerting → Rules. Three CCS alerting rules watch east from here: High Error Rate, Degraded Mode Detected, and Health Check Failure. All query cluster-east:* index patterns.
  4. When a degradation window hits, watch Alerting → Alerts. The East Microservices — High Error Rate rule fires against cluster-east:traces-apm*. The rule owner is west; the data is east.
  5. Open Discover and switch to the cluster-east filebeat data view. Search scenario: degraded. You're running a full-text log search across the east cluster's filebeat indices from west — pure CCS, no data movement.
  6. Switch to the ccs-full-analyst user in Kibana Dev Tools and run: GET cluster-east:filebeat-*/_count. The CCS coordinator (west-es01) proxies the query to east-es01 and returns the result.
Key finding: West Kibana surfaces east production data with zero replication. The NOC team sees what matters without write access to production — a clean separation of operational and engineering concerns.
Architecture benefit: CCS lets you centralise alerting and NOC dashboards in one place (west) while keeping production data isolated. A compromised NOC workstation cannot modify production indices. Query latency is typically <50ms over the net-ccs bridge.

Security Data Collection

Two lightweight Beats ship security-relevant data into east-es01, feeding the Elastic Security solution in East Kibana. Additive to the observability pipeline — existing APM, ML, SLOs, and synthetics are untouched.

Packetbeat host net
Runs on network_mode: host with NET_ADMIN / NET_RAW to sniff traffic across the net-east Docker bridges between microservices. Captures flow records plus decoded application protocols: HTTP · PostgreSQL · MySQL · MongoDB · TLS · DNS · ICMP. Ships to data streams packetbeat-* — feeds the Network section of the Security solution.
Auditbeat pid: host
File Integrity Monitoring over the shared certs volume + host binary paths. System datasets: host · login · package · process · user. Any tampering with certificates or unexpected process/user activity surfaces in auditbeat-* — feeds the Hosts section of the Security solution and correlates with Packetbeat network events on Timeline.

What this demo covers

  • Two-cluster architecture — East is the production cluster where all observability data for the refueling ops platform lives. West is the operations/NOC cluster that reads east data via CCS and hosts alerting rules.
  • Cross-Cluster Search (CCS) Basic — West Kibana queries cluster-east:* indices. Three cross-cluster data views: APM traces, Filebeat logs, Heartbeat uptime.
  • Cross-Cluster Alerting Platinum — Three rules in west Kibana watch east production data: high error rate, degraded mode detection, health check failures.
  • APM & Distributed Tracing Basic Platinum — All services ship traces to east. Distributed traces span 7 languages with up to 4-hop depth. ML anomaly detection runs on APM data (Platinum).
  • SLOs Enterprise — 14 SLOs in east Kibana: availability (99.5%) and latency SLOs for each of the 7 refueling ops services, backed by APM transaction data on a 30-day rolling window.
  • Dual Environments — Development (port 8001, anomalies enabled) and Production (port 8002, stable). Same service names, different ELASTIC_APM_ENVIRONMENT. APM data streams split by namespace.
  • Application Search Basic — Dedicated search-es01 cluster with 4 indices (aircraft, missions, airbases, crews). Powers 4 search modes in the ops gateway UI: lexical, search-as-you-type, typeahead completion, and a stored usaf-smart-search Mustache template with cross-index function scoring and <mark> highlighting.
  • Synthetic Monitoring Enterprise — Fleet Server + elastic-agent-complete on east. Three Playwright browser journeys test real refueling ops user flows.
  • Heartbeat Basic — Monitors both clusters and all microservice health endpoints. Data ships to east.
  • Filebeat + Metricbeat Basic — East filebeat autodiscovers container logs. East metricbeat collects ES, Kibana, and Docker metrics. West metricbeat: stack monitoring only.
  • Security — Network Traffic Basic Platinum — Packetbeat on host networking captures inter-service flows plus decoded HTTP, PostgreSQL, MySQL, MongoDB, TLS, and DNS. Feeds the Network section of the Elastic Security solution. Detection rules & ML jobs over this data are Platinum.
  • Security — Audit & File Integrity Basic Platinum — Auditbeat monitors the TLS cert volume + system paths for FIM, plus host, login, package, process, and user datasets. Feeds the Hosts section of the Security solution. Endpoint protection (Elastic Defend) is Platinum.
